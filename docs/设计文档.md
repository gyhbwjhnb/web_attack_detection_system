## 二、模块划分与职责

### 模块一：数据包捕获与预处理模块

**负责人**：成员 A

**职责**：
1. 使用 LibPcap/WinPcap/Npcap（Python 可使用 Scapy）从网卡实时抓取网络数据包
2. 配置 BPF 过滤器，只捕获感兴趣的流量（如 `tcp`、`udp`、`ip` 等）
3. 解析数据包各层协议头部（以太网头 → IP 头 → TCP/UDP 头），提取关键字段
4. 将原始数据包封装为统一的 `PacketInfo` 结构，传递给上层模块
5. 支持从 pcap 文件离线读取数据包（便于测试和调试）

**对外提供的能力**：
- `start_capture(interface, filter_expr)`: 开始抓包，返回一个可迭代的数据源
- `stop_capture()`: 停止抓包
- `read_pcap_file(filepath, filter_expr)`: 从 pcap 文件读取数据包
- 每捕获到一个数据包，生成一个 `PacketInfo` 对象并 yield/回调给调度层

**依赖**：公共模块（`PacketInfo` 数据结构定义）

---

### 模块二：特征匹配检测引擎模块

**负责人**：成员 B

**职责**：
1. 设计攻击特征规则的存储格式（文本文件 / JSON），至少包含：规则 ID、攻击名称、攻击类型、特征串、危险等级
2. 实现特征库的加载、更新、查询功能
3. 实现高效的特征串匹配算法（建议使用 **Aho-Corasick 多模式匹配**，避免逐条暴力匹配）
4. 对接收到的 `PacketInfo` 进行检测，检查其 payload 中是否包含特征库中的攻击特征串
5. 匹配成功时生成 `Alert` 对象
6. 支持 HTTP 协议解析，能检测 URI 解码绕过、大小写绕过等基本逃避手段（选做）

**需覆盖的攻击类型**：
- SQL 注入：`' OR 1=1`、`UNION SELECT`、`--`、`xp_cmdshell` 等
- XSS：`<script>`、`javascript:`、`onerror=` 等
- 命令注入：`; ls`、`| cat /etc/passwd`、`cmd.exe` 等
- 木马/后门：已知 C2 通信特征（如 SubSeven、特定端口连接特征）
- Web 攻击：目录遍历 `../`、`../../etc/passwd` 等

**对外提供的能力**：
- `load_signatures(filepath)`: 从文件加载攻击特征库
- `detect(packet_info) -> list[Alert]`: 对单个数据包进行特征匹配检测，返回告警列表
- `add_signature(rule)`: 动态添加一条特征规则
- `get_signature_count() -> int`: 获取当前特征库规则数量

**依赖**：公共模块（`PacketInfo`、`Alert`、`SignatureRule` 数据结构定义）

---

### 模块三：异常行为检测引擎模块

**负责人**：成员 C

**职责**：
1. 维护网络行为统计信息：按源 IP 维护连接计数器、端口访问集合、流量字节数、会话时长等
2. 建立正常行为基线（可通过配置文件设定阈值，或运行一段学习期自动设定）
3. 实现基于阈值的异常检测算法，识别以下行为：

| 异常类型 | 检测逻辑 |
|---------|---------|
| 端口扫描 | 单 IP 在时间窗口内访问目标主机的不同端口数超过阈值（如 20 个/10 秒） |
| 暴力破解 | 单 IP 在短时间内向同一服务（如 SSH:22, FTP:21, RDP:3389）发起大量连接（如 10 次/5 秒） |
| 异常外联 | 内网 IP 连接外部陌生 IP（非已知外网地址） |
| 内网横向扩散 | 内网 IP 短时间内连接多个内网其他 IP 的同一端口 |
| SYN Flood | 短时间内大量 SYN 包但无对应 ACK 完成三次握手 |

4. 检测到异常时生成 `Alert` 对象
5. 定期清理过期统计数据，防止内存膨胀

**对外提供的能力**：
- `set_baseline(config_dict)`: 设置检测阈值参数
- `analyze(packet_info) -> list[Alert]`: 对单个数据包进行统计分析并检测异常
- `get_statistics() -> dict`: 获取当前统计信息（供 GUI 面板展示）
- `reset_statistics()`: 重置所有统计计数器

**依赖**：公共模块（`PacketInfo`、`Alert` 数据结构定义）

---

### 模块四：告警输出与图形界面模块

**负责人**：成员 D

**职责**：
1. 设计并实现用户图形界面（推荐 PyQt5 或 Tkinter）
2. 实时展示告警列表（表格形式：时间、攻击类型、来源 IP、目标 IP、危险等级、详情）
3. 告警统计面板：各类攻击数量、告警趋势图
4. 告警日志管理：将告警写入日志文件（支持 txt/csv 格式），支持历史日志查看和导出
5. 系统配置界面：允许用户配置抓包网卡、过滤规则、特征库文件路径、异常检测阈值等
6. 系统启停控制：开始/停止检测的按钮

**界面布局建议**：
```
┌─────────────────────────────────────────┐
│  菜单栏：文件 | 配置 | 帮助               │
├──────────────┬──────────────────────────┤
│  控制面板     │  实时告警列表（表格）       │

│  暴力破解: 3  │                          │
│              │                          │
├──────────────┴──────────────────────────┤
│  状态栏：总告警数: 20 | 运行时间: 00:05:30 │
└─────────────────────────────────────────┘
```

**对外提供的能力**：
- `add_alert(alert)`: 接收一个 `Alert` 对象并在界面上展示
- `update_statistics(stats_dict)`: 更新统计面板数据
- 从配置界面读取用户设置，返回给主控程序使用

**依赖**：公共模块（`Alert` 数据结构定义）

## 三、公共数据结构定义

以下是各模块间共用的核心数据类，存放在 `common/` 目录下的 `data_structures.py`（Python）或对应头文件（C++）中。

### 3.1 PacketInfo —— 数据包信息

```python
from dataclasses import dataclass
from datetime import datetime

@dataclass
class PacketInfo:
    """模块一产出，模块二、三消费"""
    timestamp: datetime      # 抓包时间戳
    src_mac: str             # 源 MAC 地址
    dst_mac: str             # 目的 MAC 地址
    src_ip: str              # 源 IP 地址
    dst_ip: str              # 目的 IP 地址
    protocol: str            # 传输层协议 "TCP" / "UDP" / "ICMP"
    src_port: int            # 源端口（TCP/UDP 有效，否则为 0）
    dst_port: int            # 目的端口（TCP/UDP 有效，否则为 0）
    payload: bytes           # 应用层载荷（原始字节）
    payload_str: str         # 应用层载荷（字符串形式，不可解码部分忽略）
    payload_len: int         # 载荷长度
    flags: int               # TCP flags 位掩码（SYN/ACK/FIN/RST 等）
    seq_num: int             # TCP 序列号
    ack_num: int             # TCP 确认号
    packet_len: int          # 总包长
```

### 3.2 Alert —— 告警信息

```python
from enum import Enum

class AlertSeverity(Enum):
    LOW = 1       # 低危（信息类）
    MEDIUM = 2    # 中危（可疑行为）
    HIGH = 3      # 高危（确认攻击）
    CRITICAL = 4  # 严重（正在进行的入侵）

class AlertType(Enum):
    SIGNATURE = "signature"    # 特征匹配告警（模块二产出）
    ANOMALY = "anomaly"        # 异常行为告警（模块三产出）

@dataclass
class Alert:
    """模块二、三产出，模块四消费"""
    alert_id: int             # 告警唯一 ID（自增）
    timestamp: datetime       # 告警产生时间
    alert_type: AlertType     # 告警来源类型
    attack_type: str          # 攻击类型名称，如 "SQL注入"、"端口扫描"
    severity: AlertSeverity   # 危险等级
    src_ip: str               # 攻击来源 IP
    dst_ip: str               # 攻击目标 IP
    src_port: int             # 来源端口
    dst_port: int             # 目标端口
    description: str          # 告警详细描述
    matched_rule: str         # 匹配的规则 ID 或名称（特征匹配告警时有效）
    raw_payload: str          # 触发告警的原始载荷片段（截取前 200 字符）
```

### 3.3 SignatureRule —— 攻击特征规则

```python
@dataclass
class SignatureRule:
    """模块二使用，定义一条攻击特征规则"""
    rule_id: str              # 规则唯一 ID，如 "SQL-001"
    attack_name: str          # 攻击名称，如 "SQL注入 - OR 1=1"
    attack_type: str          # 攻击分类，如 "SQL注入"、"XSS"、"木马后门"
    pattern: str              # 特征串（支持 ASCII 和十六进制）
    severity: AlertSeverity   # 危险等级
    description: str          # 规则描述
    protocol: str             # 适用协议 "TCP"/"UDP"/"ANY"
    dst_port: int             # 目标端口（0 表示任意端口）
```

## 五、模块间调用关系与集成方式

### 5.1 主控程序（`main.py`）调度流程

```python
# main.py 主控程序（集成时由全员共同编写或由模块四负责人编写）

from module1.capture import PacketCapture
from module2.signature_engine import SignatureEngine
from module3.anomaly_engine import AnomalyEngine
from module4.gui import MainWindow
from common.data_structures import Alert
from common.utils import ConfigManager, setup_logger

def main():
    config = ConfigManager("config.json")
    logger = setup_logger("main", "logs/main.log")

    # 1. 初始化各模块
    capture = PacketCapture(config)
    sig_engine = SignatureEngine(config)
    ano_engine = AnomalyEngine(config)
    gui = MainWindow(config)

    # 2. 加载特征库
    sig_engine.load_signatures(config.get("signature.rules_file"))

    # 3. 设置异常检测基线
    ano_engine.set_baseline({
        "port_scan_threshold": config.get("anomaly.port_scan_threshold"),
        # ... 其他阈值
    })

    # 4. 启动抓包循环
    for packet_info in capture.start_capture(
        interface=config.get("capture.interface"),
        filter_expr=config.get("capture.filter")
    ):
        # 5. 检测
        alerts_sig = sig_engine.detect(packet_info)      # 特征匹配
        alerts_ano = ano_engine.analyze(packet_info)     # 异常检测

        # 6. 告警汇总
        for alert in alerts_sig + alerts_ano:
            gui.add_alert(alert)                          # 推送到界面

        # 7. 可选：定期更新统计面板
        gui.update_statistics(ano_engine.get_statistics())

if __name__ == "__main__":
    main()
```

> **注意**：各模块之间不直接相互调用。模块一将 `PacketInfo` 输出给调度层，调度层分发给模块二和模块三，两个检测模块各自产出 `Alert` 汇总给模块四。这样保持了模块的独立性。

## 八、关键约定

1. **数据传递**：所有模块间数据传递使用 `common/data_structures.py` 中定义的数据类，不允许使用原始字典或元组。
2. **异常处理**：每个模块应自行处理内部异常，不允许异常向上抛出导致整个系统崩溃。对外接口返回空列表而非 `None`。
3. **日志规范**：所有模块统一使用 `common/utils.py` 中的 `setup_logger` 获取 logger，日志级别建议：模块一/四用 `INFO`，模块二/三检测到攻击时用 `WARNING`。
4. **配置统一**：所有可配置参数通过 `ConfigManager` 从 `config.json` 读取，不允许在代码中硬编码阈值或路径。
5. **编码规范**：Python 使用 UTF-8 编码，注释和标识符使用英文（若小组协商同意也可使用中文注释）。缩进 4 空格。
6. **版本控制**：建议使用 Git，各成员在自己分支上开发各自模块，主分支只用于集成。

## 九、智能检测增强（v2.0 新增）

### 9.1 检测器插件化架构

模块三重构为插件化架构，通过 `IDetector` 抽象接口实现检测器的热插拔：

```python
class IDetector:
    name: str              # 插件唯一标识
    category: str          # 分类（scan / brute / exfil / baseline / enhance）
    priority: int          # 执行顺序（越小越先）
    enabled: bool          # 开关

    def process(record, now) -> List[Alert]:        # 核心检测
    def post_process_batch(alerts) -> List[Alert]:  # 可选：后处理增强
```

**运行时热插拔 API**：

```python
engine.add_detector(MyDetector())     # 动态加载新检测器
engine.remove_detector("port_scan")   # 动态移除检测器
engine.list_detectors()               # 列出全部检测器
engine.get_detector("name")           # 获取特定检测器实例
```

新增自定义检测器只需 3 步：创建文件实现 IDetector → 在 `__init__.py` 注册 → 重启生效。

### 9.2 规则热加载

模块二支持文件夹级别的规则热加载：

- `load_rules_with_hot_reload()` 监听 `data/rules/` 目录
- 后台线程每 3 秒检测文件修改，自动重载所有规则
- 支持多文件分散管理（`signatures.json` + `custom_sql.json` + ...）
- GUI 提供"导入规则"按钮（工具栏 + 配置菜单），选择 JSON 文件自动复制到规则目录并立即生效

### 9.3 攻击置信度评分（Confidence Scorer）

后处理阶段为每条告警动态计算 0.0-1.0 置信度评分：

| 攻击类型 | 评分依据 |
|---------|---------|
| 端口扫描 | 目标端口数 + 扫描速率 + 端口连续度 |
| 暴力破解 | 连接密度 + 高关注端口加分（22/3389/3306 等） |
| 异常外联 | 目标稀有度 + 恶意端口 + 非标准端口 |
| SYN Flood | SYN 密度偏离基线倍率 |
| 带宽异常 | 偏离基线流量倍数 |
| 横向扩散 | 接触主机数 + 跨子网检测 |
| 行为突变 | 偏离标准差倍数 |

所有告警保留不抑制（含低分告警），仅补充置信度字段供 GUI 参考。

### 9.4 异常时段检测（Time Profile）

基于 24 小时 EMA 基线的时段异常检测：

- 维护每主机 `{0..23: (均值, 方差, 样本数)}` 小时直方图
- 凌晨 0-6 点连接数 > 历史均值 + 2σ → 自动提升告警置信度（最多 +0.25）
- 告警描述追加 `[夜间异常时段]` 标记
- EMA 持续更新，适应网络环境变化

### 9.5 自学习误报抑制（Suppression Learner）

从用户反馈中自动学习误报模式：

```
GUI 右键标记"误报" → 提取 (src_ip, attack_type, dst_port) 签名
                   → 持久化到 data/suppression_rules.json
                   → 同模式标记 ≥2 次后，后续匹配告警自动 -0.35 置信度
```

- 跨会话保留抑制规则
- 支持精确匹配 + 类型泛化（忽略端口）两类抑制策略
- 告警匹配抑制规则时追加 `user_suppressed` 标签
