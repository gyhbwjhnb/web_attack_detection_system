## 1. 项目概述

### 1.1 项目定位

网络攻击检测系统 (NIDS) 是一款**轻量级、可扩展的双引擎网络入侵检测系统**，基于特征匹配（已知攻击签名）和异常行为分析（行为基线模型）两条检测路径并行工作，通过统一的消息总线将结果汇聚到图形化界面。

### 1.2 设计目标

| 目标 | 实现方式 |
|------|---------|
| **双引擎检测** | 模块2 特征匹配 + 模块3 异常行为分析并行处理 |
| **插件化扩展** | IDetector 接口使检测器完全独立，新增无需改引擎 |
| **零外部依赖 GUI** | 使用 Python 内置 tkinter，无需 PyQt5/WxPython |
| **离线可用** | 支持 PCAP 回放和模拟数据注入，无需 Npcap |
| **规则热加载** | 拖放 JSON 规则文件到 data/rules/ 目录即生效 |
| **智能告警** | 置信度评分、时段检测、自学习误报抑制 |

### 1.3 技术栈

```
Python 3.8+ (纯标准库)
├── tkinter          GUI 框架（内置）
├── scapy            网络抓包与协议解析（唯一的第三方依赖）
├── threading        多线程（抓包、检测、GUI 各一线程）
├── json             规则与配置存储
├── logging          统一日志输出
└── re               正则匹配（特征规则）
```

### 2.2 设计原则

1. **松耦合**：模块间仅通过 MessageBus 通信，无直接依赖
2. **单一职责**：每个模块只做一件事
3. **开闭原则**：对扩展开放（新增检测器/规则），对修改封闭（不动引擎核心）
4. **接口隔离**：每个模块暴露最小化 API

## 3. 模块详细设计

### 3.1 模块一：数据包捕获与预处理

| 文件 | 职责 | 核心 API |
|------|------|---------|
| `capture.py` | 3 种数据源：在线网卡抓包、离线 PCAP 回放、内存模拟注入 | `start()`, `stop()`, `set_mode()` |
| `packet_parser.py` | scapy 包 → 统一 TrafficRecord（逐层解析 Ether→IP→TCP/UDP→HTTP/DNS/TLS/ICMP/ARP） | `parse_packet()`, `create_fake_http_record()`, `create_fake_dns_record()` |

**支持的协议层**：

```
Ethernet → IPv4/IPv6 → TCP/UDP/ICMP → HTTP / DNS / TLS / ARP / SMB
```

**关键设计决策**：
- 解析器为无状态纯函数，可并行调用
- 模拟模式为演示和测试而设计，无需安装 Npcap
- 所有输出统一为 `TrafficRecord` 数据类，通过 MessageBus 发布

### 3.2 模块二：特征匹配检测引擎

| 文件 | 职责 | 核心 API |
|------|------|---------|
| `signature_engine.py` | 规则加载/热加载、三级过滤（协议→端口→正则）、暴力破解频次统计、告警去重 | `load_rules()`, `load_rules_with_hot_reload()`, `process_traffic()` |
| `matcher.py` | Aho-Corasick 多模式匹配算法实现（Trie + BFS Failure Link） | `add_pattern()`, `search()`, `clear()` |
| `integration.py` | 一行 `connect(engine)` 完成 MessageBus 订阅/发布 | `connect()`, `disconnect()` |

**检测流程**：

```
TrafficRecord → 提取检测文本（HTTP全字段→payload）
             → Aho-Corasick 多模式匹配（一次扫描命中所有规则）
             → 协议/端口过滤
             → 分类开关过滤
             → 暴力破解频次检测（额外处理）
             → 告警去重（同源同目的同类型窗口期去重）
             → 发布 signature_alert 事件
```

**规则结构** (`data/signatures.json`):

```json
{
  "rule_id": "SQL-001",
  "attack_name": "SQL注入 - UNION SELECT",
  "attack_type": "sql_injection",
  "pattern": "(?i)union.*select",
  "severity": 4,
  "description": "检测到UNION SELECT型SQL注入攻击",
  "protocol": "TCP",
  "dst_port": 80
}
```

**热加载机制**：
- `data/rules/` 目录后台线程 3 秒轮询
- 检测到 `.json` 文件变更后自动调用 `load_rules()`
- 路径自动解析为项目根目录，不受工作目录影响

### 3.3 模块三：异常行为检测引擎

| 文件 | 职责 | 核心 API |
|------|------|---------|
| `anomaly_engine.py` | EMA 基线学习、滑动窗口统计、检测器注册/移除/列表、攻击链构建 | `process_traffic()`, `add_detector()`, `remove_detector()`, `list_detectors()`, `get_detector()` |

**10 个检测器插件**:

| 优先级 | 文件名 | 检测器 | 检测类型 |
|:---:|------|------|------|
| 10 | `port_scan.py` | PortScanDetector | 端口扫描 |
| 15 | `syn_flood.py` | SynFloodDetector | SYN Flood / DDoS |
| 18 | `bandwidth_anomaly.py` | BandwidthAnomalyDetector | 带宽突发异常 |
| 20 | `brute_force.py` | BruteForceDetector | 暴力破解（敏感端口加分） |
| 25 | `abnormal_outbound.py` | AbnormalOutboundDetector | 异常外联陌生 IP |
| 30 | `lateral_movement.py` | LateralMovementDetector | 内网横向扩散（跨子网） |
| 40 | `behavioral_deviation.py` | BehavioralDeviationDetector | 行为突变（4维度） |
| 50 | `time_profile.py` | TimeProfileDetector | 24h 异常时段 EMA |
| 80 | `suppression_learner.py` | SuppressionLearner | 自学习误报抑制 |
| 85 | `confidence_scorer.py` | ConfidenceScorer | 置信度后处理评分 |

**执行时序**：

```
TrafficRecord
  → 基线更新（EMA 自适应）
  → 按 priority 排序遍历 8 个主检测器 (10-50)
     每个检测器: set_context() → process() → 发布 anomaly_alert
  → 后处理阶段 (80-85)
     SuppressionLearner.post_process_batch()
     ConfidenceScorer.post_process_batch()
  → 攻击链构建
```

**EMA 基线模型**：

```
新基线 = α × 当前观测值 + (1-α) × 旧基线
α = 2 / (N + 1)，N ≈ 60（约 2 分钟窗口）

每个主机维护:
  - avg_connections      平均连接数
  - avg_unique_peers     平均对端 IP 数
  - avg_syn_ratio        平均 SYN 比例
  - avg_bandwidth        平均带宽
  - hour_baselines[24]   24h EMA 时段基线（time_profile）
```

## 4. 数据流与 MessageBus

### 4.1 核心事件

| 事件名 | 发布者 | 订阅者 | 数据结构 |
|--------|-------|--------|---------|
| `traffic_record` | 模块1 | 模块2, 模块3, 模块4 | `TrafficRecord` |
| `signature_alert` | 模块2 | 模块4 | `Alert` |
| `anomaly_alert` | 模块3 | 模块4 | `Alert` |
| `attack_chain` | 模块3 | 模块4 | `AttackChain` |
| `statistics` | 模块3 | 模块4 | `dict` |
| `config_change` | 模块4 | 模块2, 模块3 | `dict` |

### 4.2 完整数据流路径

```
1. 模块1 capture.py → parse_packet() → TrafficRecord
2. MessageBus.publish("traffic_record", record)
3. 模块2 integration.py 接收 → process_traffic() → Alert
4. 模块3 anomaly_engine.py 接收 → 10个检测器遍历 → Alert
5. MessageBus.publish("signature_alert" / "anomaly_alert", alert)
6. 模块4 main_window.py 接收 → GUI 表格更新
```

## 6. 智能增强特性 (v2.0)

### 6.1 置信度评分 (S6)

**设计动机**：同是"端口扫描"告警，扫描 3 个端口 vs 1000 个端口的恶意程度完全不同。

**评分算法**：

| 攻击类型 | 评分公式 |
|---------|---------|
| port_scan | `min(1.0, 0.3 + 端口数×0.01 + 速率×0.05 + 端口连续性加分)` |
| brute_force | `min(1.0, 0.3 + 密度×0.1 + (端口 in {22,3389,3306} ? 0.2 : 0))` |
| syn_flood | `min(1.0, 0.3 + (SYN数/基线)×0.05)` |
| abnormal_outbound | `min(1.0, 0.3 + 端口恶意加分 + 非标准端口加分 + 陌生IP加分)` |
| lateral_movement | `min(1.0, 0.3 + 主机数×0.03 + 跨子网加分)` |
| bandwidth_anomaly | `min(1.0, 0.3 + (当前/基线-1)×0.2)` |
| behavioral_deviation | `min(1.0, 0.3 + 最大维度偏离×0.2)` |

**实现**：ConfidenceScorer (priority=85)，在所有检测器之后执行 `post_process_batch()`。

### 6.2 异常时段检测 (S7)

**设计动机**：凌晨 3 点的 SSH 连接远比下午 3 点的可疑。

**实现**：TimeProfileDetector (priority=50)，每主机维护 `{小时: (EMA均值, EMA方差, 样本数)}` 24 格直方图：

```
告警条件: 当前时段连接数 > EMA 均值 + 2 × EMA 标准差
置信度提升: +0.15（偏离2σ）→ +0.25（偏离5σ）
告警标记: 描述追加 "[夜间异常时段(03:00)]"
仅触发于: 0:00-6:00
```

### 6.3 自学习误报抑制 (S8)

**设计动机**：用户手动标记的误报应该被系统记住并自动处理。

**流程**：

```
GUI 右键 → "误报"
  → main_window._set_alert_status()
  → on_alert_ignored 回调
  → SuppressionLearner.on_user_ignore()
  → 记录签名: "src_ip|attack_type|dst_port"
  → 持久化到 data/suppression_rules.json
  → 同签名累计 2 次标记 → 后续匹配告警自动 -0.35 置信度
```

**跨会话持久化**：抑制规则以 JSON 格式保存到 `data/suppression_rules.json`，重启后自动加载。

## 7. 可扩展性设计

### 7.1 IDetector 插件接口

```python
class IDetector(ABC):
    name: str = ""         # 唯一标识
    category: str = ""     # 分类标签
    priority: int = 50     # 执行顺序（越小越先）
    enabled: bool = True   # 开关

    def set_context(self, hosts, baselines, config, lock, count_recent, make_alert):
        """引擎注入共享上下文"""

    def process(self, record: TrafficRecord, now: float) -> List[Alert]:
        """核心检测方法"""

    def post_process_batch(self, alerts: List[Alert]) -> List[Alert]:
        """后处理：在所有检测器跑完后统一处理（可选）"""
```

### 7.2 新增检测器步骤

```
1. 在 detectors/ 下新建文件 → class MyDetector(IDetector)
2. 在 __init__.py 的 _ALL_DETECTORS 注册
3. 重启 → 自动注入 hosts/baselines/config/lock，按 priority 排序
```

无需修改 `anomaly_engine.py`。

### 7.3 新增自定义规则

在 `data/rules/` 下放置 `.json` 文件 → 3 秒内自动热加载 → 控制台输出 `"检测到规则变更，已重新加载 N 条规则"`。

## 11. 与 Snort3 的对比分析

| 维度 | Snort3 | 本项目 NIDS |
|------|--------|-----------|
| **语言** | C++ | Python |
| **性能** | 高速包处理（DPDK 支持） | 适中（适合教育/实验环境） |
| **架构** | 插件化（Codec/IpsOption/Inspector 等接口） | 检测器插件化（IDetector 接口 + MessageBus） |
| **规则语法** | 自有 DSL（alert tcp ... msg:... sid:...） | JSON 格式（可读性强，热加载） |
| **规则热加载** | 需 `snort -HUP` 信号 | 目录监控自动重载，3 秒内生效 |
| **模式匹配** | 专有算法（经高度优化） | Aho-Corasick + URL 解码对抗 |
| **异常检测** | 无内置 EMA 行为基线 | **内置 EMA 自适应基线 + 10 个插件检测器** |
| **置信度评分** | 无 | **7 种攻击类型动态 0.0-1.0 评分** |
| **误报抑制** | 阈值抑制（Suppress/Threshold） | **自学习误报抑制（用户标记 + 跨会话持久化）** |
| **时段感知** | 无 | **24h EMA 时段基线，凌晨异常自动提权** |
| **攻击链** | 无 | **基于 Cyber Kill Chain 的攻击阶段映射** |
| **GUI** | 第三方（Snort3 无自带 GUI） | 内置完整 tkinter GUI |
| **数据源** | 在线抓包 + PCAP | 在线抓包 + PCAP + 模拟数据（3 种） |
| **白名单** | 配置复杂（需重载规则） | GUI 即时添加，无需重启 |
| **报告导出** | 日志文件（需外部工具分析） | HTML 报告（含分布图 + 明细表）+ CSV/TXT |
| **启动门槛** | 需安装 Npcap + 编译或安装 | 模拟/离线模式无需 Npcap |
| **安装包大小** | ~20MB（运行时） | ~15MB（exe） |
| **学习曲线** | 陡峭（复杂配置语法） | 平缓（Python + JSON + 点 GUI） |

**核心差异化优势**：

1. **异常检测 + 基线学习**：Snort3 是纯特征匹配引擎，本项目具备 EMA 自适应行为基线模型
2. **智能告警增强**：置信度评分 + 时段感知 + 自学习误报抑制，Snort3 不具备
3. **开箱即用 GUI**：Snort3 无自带 GUI，需要 Barnyard2/SGUIL/ELK 等第三方工具
4. **规则热加载**：目录级文件监控即生效，比 Snort3 的 `kill -HUP` 更友好
5. **零编译运行**：纯 Python 脚本，改动即时生效，适合教学和快速原型迭代
6. **可扩展性**：IDetector 接口使新增检测器仅需 3 步
