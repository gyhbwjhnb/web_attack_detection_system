================================================================================
  第三章  系统总体架构
================================================================================

3.1 总体架构描述

本系统采用单主机部署的单体四模块架构，通过 MessageBus（发布-订阅消息总线）实现模块间松耦合通信。
四个模块分别是：数据包捕获与预处理（模块1）、特征匹配检测引擎（模块2）、异常行为检测引擎（模块3）、
图形用户界面（模块4）。

【四模块在系统中的位置与关系】

模块1（数据包捕获与预处理）是整个系统的唯一数据入口，负责从物理网卡、PCAP 文件或内存模拟三种
数据源获取网络流量，经过协议解析后将其转换为统一的 TrafficRecord 结构体，通过 MessageBus 发布。

模块2（特征匹配检测引擎）和模块3（异常行为检测引擎）是并列的两个检测引擎，同时订阅模块1发布的
traffic_record 事件。两者独立并行工作，互不依赖：

  ● 模块2（特征匹配）基于已知攻击特征库（signatures.json），采用"协议过滤 → 端口过滤 → 正则匹配"
    三级过滤机制识别已知攻击类型（SQL注入、XSS、命令注入等 20+ 类），输出 signature_alert。

  ● 模块3（异常检测）基于行为基线模型，通过 60 秒学习期建立正常网络行为基线，然后运行六大检测器
    （端口扫描、SYN Flood、异常外联、行为突变、带宽异常、横向扩散）实时对比当前行为与基线偏差，
    发现未知威胁，输出 anomaly_alert。同时具备 Cyber Kill Chain 攻击链构建能力。

模块4（GUI 图形界面）是系统的终端展示层，同时订阅 traffic_record、signature_alert 和
anomaly_alert 三类事件，将所有数据统一汇聚到界面上展示。模块4 是纯粹的消息消费者，不区分告警
来源（模块2 还是 模块3），确保引擎层的任何修改不影响界面层。

【数据流总体描述】

系统运行时的完整数据流如下（文字版）：

  步骤1：数据进入
      模块1 从以下三种来源之一获取原始数据包：
        来源A：物理网卡（通过 Npcap/libpcap 驱动，在线模式）
        来源B：PCAP 文件（离线回放模式）
        来源C：内存模拟数据（测试演示模式）

  步骤2：协议解析（模块1 内部）
      原始数据包进入模块1 的子模块2（packet_parser.py），逐层解析：
        链路层（MAC 地址）→ 网络层（IP、TTL、协议类型）→ 传输层（TCP/UDP/ICMP 端口和标志位）
        → 应用层（HTTP/DNS/TLS/ARP 深度解析）
      输出：结构化的 TrafficRecord 对象（约 60 个字段）

  步骤3：数据分发
      模块1 通过 MessageBus.publish("traffic_record", record) 将 TrafficRecord 发布到消息总线。
      同时通过内部回调机制更新统计（包数、字节数、流数、协议分布）。

  步骤4a：已知攻击检测（模块2 并行处理）
      模块2 收到 TrafficRecord → 白名单过滤 → 三级过滤（协议→端口→正则）→ 匹配成功则生成 Alert
      → MessageBus.publish("signature_alert", alert)

  步骤4b：未知威胁检测（模块3 并行处理，与步骤4a 同时进行）
      模块3 收到 TrafficRecord → 白名单过滤 → 基线更新（上轨） + 异常判定（下轨，6 大检测器）
      → 异常则生成 Alert → MessageBus.publish("anomaly_alert", alert)
      → 攻击链构建器将告警串联为 Kill Chain

  步骤5：结果展示（模块4 汇聚）
      模块4 同时接收三类 MessageBus 消息：
        ● traffic_record  → 更新流量表格 + 刷新筛选建议
        ● signature_alert → 插入告警表格 + 更新统计面板
        ● anomaly_alert   → 插入告警表格 + 更新统计面板
      用户在 GUI 中可进行筛选、标记状态、编辑备注、导出报告、配置白名单/阈值等操作

【系统技术栈分层描述】

系统技术栈分为三层：

  数据层：
    ● scapy 库 + Npcap/libpcap 驱动（Windows 在线抓包）或 libpcap（Linux/macOS）
    ● scapy.rdpcap() 离线 PCAP 解析
    ● 内存模拟数据构造函数（create_fake_*）

  引擎层：
    ● 签名引擎：Python 标准 re 模块（预编译正则表达式）
    ● 异常引擎：Python 标准库（deque 滑动窗口、EWMA 指数滑动平均、字典/集合）
    ● 攻击链：Cyber Kill Chain 七阶段模型

  展示层：
    ● tkinter 标准库 GUI（ttk.Treeview、ttk.Notebook）
    ● 内嵌 CSS 的 HTML 报告生成（无外部依赖）
    ● root.after(0, ...) 线程安全机制

【MessageBus 通信机制】

MessageBus 是本系统的核心通信中间件，采用发布-订阅（Pub-Sub）模式：

  事件类型（5 种）：
    EVENT_TRAFFIC_RECORD   — 模块1 发布，模块2/3/4 订阅
    EVENT_SIGNATURE_ALERT  — 模块2 发布，模块4 订阅
    EVENT_ANOMALY_ALERT    — 模块3 发布，模块4 订阅
    EVENT_STATISTICS       — 模块1 发布，模块4 订阅（抓包统计）
    EVENT_CONFIG_CHANGE    — 模块4 发布，模块2/3 订阅（配置变更通知）

  通信特点：
    ● 松耦合：发布者和订阅者通过事件类型关联，互不感知对方的存在
    ● 一对多：一个事件可被多个订阅者同时接收
    ● 线程安全：消息回调在发布者线程中同步执行，GUI 侧通过 root.after 投递到主线程

================================================================================
  第四章  系统功能模块1 — 数据包捕获与预处理
================================================================================

4.1 模块1 内部结构与子模块协作关系

模块1（数据包捕获与预处理）由两个子模块构成：capture.py（抓包引擎）和 packet_parser.py（协议解析器）。
两个子模块通过 "一个原始包 → 一次解析调用 → 一条 TrafficRecord" 的管道式协作完成数据处理。

【数据流全景】

数据从三个来源进入系统：在线网卡（Npcap 驱动）、离线 PCAP 文件、内存模拟构造。无论哪种来源，
最终都汇聚到同一个处理入口——_handle_packet() 方法，经过协议解析后以统一的 TrafficRecord 格式
通过 MessageBus 或回调函数输出。

【子模块1: capture.py — 抓包引擎】

该子模块是系统唯一的数据入口，负责三个职责：获取原始数据包、调度解析流程、分发解析结果。

核心逻辑流程如下：

  (1) 初始化阶段
      ● 用户调用 start() 方法，传入 interface（网卡名）或 offline_pcap（文件路径）
      ● 在线模式：引擎尝试导入 scapy，自动选择最优物理网卡（过滤虚拟适配器）
      ● 离线模式：引擎检查 PCAP 文件是否存在，存在则打开
      ● 模拟模式：无需初始化，直接调用 create_fake_*() 工具函数

  (2) 在线抓包循环（在线模式）
      ● 引擎在后台线程中调用 scapy.sniff()，开启混杂模式抓包
      ● 每个到达网卡的数据包触发一次回调 → _handle_packet(packet)
      ● _handle_packet 内部执行三步管线：
          a) 调用 packet_parser.parse_packet(packet) → 得到 TrafficRecord 或 None
          b) 更新内部统计：包计数、字节计数、协议分布、TCP/UDP 流去重
          c) 调用 _dispatch(record) → 遍历所有注册的回调函数 + 通过 MessageBus 发布

  (3) 离线解析循环（离线模式）
      ● 引擎在主线程中调用 scapy.rdpcap() 一次性读取全部数据包
      ● 对每个包依次执行 _handle_packet()，处理流程与在线模式完全相同
      ● 所有包处理完毕后 _running 自动置为 False

  (4) 停止阶段
      ● 用户调用 stop() → 设置 threading.Event 停止标志
      ● 后台嗅探线程检测到停止信号后退出
      ● 主线程等待嗅探线程最多 3 秒后强制返回

  (5) 关键设计决策
      ● 双重输出通道：同时支持回调函数（set_on_traffic_callback）和 MessageBus 发布，
        使模块1可以独立运行（仅回调）或集成到系统中（MessageBus）
      ● 网卡自动选择算法：按优先级排序（物理有线 > 物理无线 > 虚拟网卡），
        自动排除 VMware/VirtualBox/Hyper-V/Docker/WSL/Npcap Loopback 等虚拟适配器
      ● 流统计去重：基于五元组（src_ip:port-dst_ip:port-protocol）的集合维护，
        支持 TCP 和 UDP 流的独立计数

【子模块2: packet_parser.py — 协议解析器】

该子模块是一组无状态纯函数，输入为单个 scapy 数据包对象，输出为结构化的 TrafficRecord。

解析层次自底向上，共五层：

  (1) 第一层：入口与分流 — parse_packet() 主函数
      ● 检查数据包是否包含 ARP 层 → 是则调用 _parse_arp()，独立解析 ARP 请求/应答
      ● 检查数据包是否包含 IP 层（IPv4 或 IPv6）→ 否（如 PPPoE）则返回 None，静默跳过
      ● 包含 IP 层 → 创建 TrafficRecord 空对象，进入逐层填充

  (2) 第二层：链路层解析 — _parse_ethernet()
      ● 从 scapy Ether 层提取源 MAC 地址和目的 MAC 地址
      ● 写入 record.src.mac 和 record.dst.mac

  (3) 第三层：网络层解析 — _parse_ip()
      ● 从 scapy IP 层提取源 IP、目的 IP、TTL、IP 版本
      ● 判断上层协议类型：检查 TCP/UDP/ICMP 层的有无 → 设置 record.protocol
      ● 写入 record.src 和 record.dst 的 IPEndpoint（包含 ip、port、is_internal 自动判断）

  (4) 第四层：传输层解析 — _parse_transport()
      ● TCP 包：提取源端口、目的端口、标志位（如 0x02=SYN, 0x18=PSH+ACK）、序列号、确认号
      ● UDP 包：提取源端口、目的端口
      ● ICMP 包：提取 ICMP 类型（8=Echo Request / 0=Echo Reply）和代码

  (5) 第五层：应用层深度解析
      ● HTTP 检测与解析 (_parse_http)：
          - 从 payload 首行判断请求（GET/POST/...）或响应（HTTP/1.1 200）
          - 提取 method、URI（自动 URL 解码）、Host、User-Agent、Referer、Body
          - 构造 http_query_params：自动将 URI 中 ?k=v&k=v 解析为字典
          - 构造 all_http_text：将 URI、Body、Headers 拼接为统一文本供模块2特征匹配

      ● DNS 检测与解析 (_parse_dns)：
          - 从 payload_raw 原始字节解析 DNS 头部、查询域名、查询类型（A/AAAA/MX/TXT）
          - 解析响应答案列表（A 记录的 IPv4、AAAA 记录的 IPv6、CNAME 别名）
          - 支持 DNS 压缩指针展开（域名压缩编码 0xC0）

      ● TLS 检测与 SNI 提取 (_detect_tls)：
          - 检查 payload_raw 首字节是否为 0x16（TLS Handshake）
          - 从 ClientHello 消息中遍历扩展字段，找到 type=0x0000（server_name）
          - 提取 SNI 域名明文（加密前的目标服务器域名）

      ● ARP 解析 (_parse_arp)：
          - 提取操作类型（1=Request / 2=Reply）
          - 提取发送方 IP、发送方 MAC、目标 IP、目标 MAC
          - 生成流 ID 格式："ARP:<src_ip>-><dst_ip>"

      ● 端口标记检测：
          22→SSH, 21→FTP, 25→SMTP, 445/139→SMB, 3389→RDP, 3306→MySQL,
          6379→Redis, 27017→MongoDB

  (6) 输出阶段
      ● 生成流 ID：格式为 "src_ip:src_port-dst_ip:dst_port-协议名"（五元组标识）
      ● 计算 payload_size
      ● 返回完整的 TrafficRecord 对象（约 60 个字段）

【两个子模块的协作方式】

子模块1 (capture.py) 和子模块2 (packet_parser.py) 之间是严格的单向调用关系：

  capture._handle_packet(raw_packet)
      │
      ├──→ packet_parser.parse_packet(raw_packet)  [调用子模块2]
      │         │
      │         └──→ 返回 TrafficRecord 或 None
      │
      ├──→ 更新 capture 内部统计（包数、字节数、流数、协议分布）
      │
      └──→ capture._dispatch(record)  [分发结果]
               │
               ├──→ 遍历 _on_traffic_callbacks 列表，逐个回调
               └──→ MessageBus.publish("traffic_record", record)

该协作是同步的：每个原始包到达时，解析 → 统计 → 分发是一条不可分割的管线。
但在线模式下，scapy.sniff() 在独立线程中运行，整条管线发生在后台，不阻塞 GUI 主线程。


================================================================================
  第五章  系统功能模块2 — 特征匹配检测引擎
================================================================================

5.1 模块2 内部结构与子模块协作关系

模块2（特征匹配检测引擎）由两个子模块构成：signature_engine.py（签名引擎）和 matcher.py（规则匹配器）。
两个子模块通过 "检测流程调度 → 逐规则匹配判断" 的分层协作完成已知攻击检测。

【数据流全景】

模块2 通过 MessageBus 订阅模块1发布的 "traffic_record" 事件。每条 TrafficRecord 到达后，
先经过白名单过滤，再进入规则匹配循环——遍历所有已加载的攻击特征规则，每条规则依次经过
协议过滤、端口过滤、正则匹配三级判断。任一规则匹配成功即生成 Alert，通过 MessageBus 发布
"signature_alert" 供模块4 GUI 展示。

【子模块1: signature_engine.py — 签名引擎】

该子模块是模块2的控制中心，负责四个职责：规则加载、检测调度、告警生成、统计管理。

核心逻辑流程如下：

  (1) 初始化与规则加载
      ● 引擎启动时调用 load_rules() → 读取 data/signatures.json 文件
      ● 使用 SignatureRule.from_dict() 将 JSON 规则转为 Python 对象
      ● 每条规则的正则表达式 pattern 预编译为 re.Pattern 对象（存放于 _compiled_rules）
      ● JSON 解析失败或文件缺失 → 使用内置默认规则集（包含最常见的 5-6 种攻击）
      ● 返回加载成功的规则总数

  (2) 检测入口 — process_traffic(record) 方法
      每条 TrafficRecord 到达时执行以下流程：

      a) 白名单过滤
         ● 检查 record.src.ip 是否在 WHITELIST_IPS 中
         ● 检查 record.dst.ip 是否在 WHITELIST_IPS 中
         ● 任一命中 → 直接返回 []（不产生任何告警，但流量仍计入统计）

      b) 规则遍历循环
         ● 获取扫描文本：优先使用 record.all_http_text（HTTP 协议时），否则使用 record.payload
         ● 遍历 _compiled_rules 列表中所有启用的规则
         ● 对每条规则调用 matcher.match(record, rule) → 返回 (matched, value)

      c) 三级匹配机制（在 matcher.py 中实现，此处调度调用）
         第一级——协议过滤：规则指定 protocol="HTTP"，但 record 是 TCP → 跳过
         第二级——端口过滤：规则指定 dst_port=80，但 record.dst.port=443 → 跳过
         第三级——正则匹配：re.search(compiled_pattern, text) → 匹配成功/失败

      d) 告警生成
         ● 某条规则匹配成功 → 创建 Alert 对象，填充以下字段：
             - alert_id：12 位 UUID
             - attack_type：规则中的 attack_type（如 "sql_injection"）
             - attack_name：规则中的 attack_name（如 "SQL注入 - OR 1=1"）
             - severity：规则中的默认严重度（1-5）
             - confidence：匹配置信度（精确匹配=0.95）
             - rule_id：触发的规则 ID
             - src_ip / src_port / dst_ip / dst_port：从 record 提取
             - matched_pattern：实际匹配到的内容
             - payload_snippet：触发告警的载荷片段（截取前后各 50 字符）
             - suggestion：处理建议（由 get_attack_info 从配置获取）

      e) 告警发布
         ● 调用 MessageBus.publish("signature_alert", alert)
         ● update 内部统计：_total_alerts += 1, _alerts_by_type[attack_type] += 1

  (3) 去重机制
      ● 维护 _recent_alerts 字典：{(attack_type, src_ip, dst_ip): 最后告警时间}
      ● 新告警与历史告警的 (attack_type, src_ip, dst_ip) 三元组相同且时间差 < 5秒 → 去重
      ● 去重的告警不计入 _total_alerts 但增加原告警的 repeat_count

  (4) 统计输出 — get_statistics()
      ● 返回字典包含：total_alerts（总告警数）、alerts_by_type（按类型分布）、
        rules_loaded（加载规则数）、traffic_processed（处理流量总数）

【子模块2: matcher.py — 规则匹配器】

该子模块是无状态的纯函数，输入一条 TrafficRecord 和一条 SignatureRule，
输出匹配结果 (matched: bool, matched_value: str)。

三级过滤逻辑详解：

  (1) 协议过滤（第一级）
      ● 读取 rule.protocol 字段（如 "HTTP", "TCP", "DNS"，空字符串=不限）
      ● 若 rule.protocol 非空且与 record.protocol.value 不匹配 → 直接返回 (False, "")
      ● 设计意图：CSV/JSON 注入这类规则限定 protocol="HTTP"，避免在非 HTTP 流量中误报

  (2) 端口过滤（第二级）
      ● 读取 rule.dst_port 字段（整数，0=不限端口）
      ● 若 rule.dst_port > 0 且不等于 record.dst.port → 直接返回 (False, "")
      ● 设计意图：MySQL 攻击规则限定 dst_port=3306，避免在 80 端口的普通流量中误报

  (3) 正则匹配（第三级）
      ● 获取扫描文本：根据协议类型智能选择
          - HTTP 协议 → record.all_http_text（拼接 URI + Body + Headers + Referer + UA）
          - DNS 协议 → record.dns_query（仅匹配域名）
          - 其他协议 → record.payload
      ● 对扫描文本执行 re.search(rule.compiled_pattern, text)
      ● 匹配成功 → 返回 (True, match.group(0))（即匹配到的具体字符串）
      ● 匹配失败 → 返回 (False, "")

  (4) 预编译优化
      ● 所有规则的正则表达式在加载时（load_rules 阶段）编译为 re.Pattern 对象
      ● 编译后的对象存储在 rule.compiled_pattern 字段
      ● 运行时不再有编译开销，100 条规则遍历 < 1ms

【两个子模块的协作方式】

子模块1 (signature_engine.py) 和子模块2 (matcher.py) 之间是调用者-被调用者的关系：

  signature_engine.process_traffic(record)
      │
      ├── 白名单检查 → 命中则 return []
      │
      ├── 获取扫描文本: record.all_http_text
      │
      ├── for rule in self._rules:  ← 外层循环（引擎调度）
      │       │
      │       ├── matcher.match(record, rule)  ← 内层调用（匹配器判断）
      │       │       │
      │       │       ├── 协议过滤 → 不匹配则 return (False, "")
      │       │       ├── 端口过滤 → 不匹配则 return (False, "")
      │       │       └── 正则匹配 → 返回 (matched, matched_value)
      │       │
      │       ├── matched == True → 创建 Alert 并入去重检查
      │       └── matched == False → continue 下一条规则
      │
      └── for 循环结束后 → 返回 alerts 列表

该协作是同步且顺序的：一条 record 依次经过所有规则，任一规则命中即产生告警，
同一条流量可能触发多条不同规则（如同时检测到 SQLI + XSS），产生多个告警。


================================================================================
  第六章  系统功能模块3 — 异常行为检测引擎
================================================================================

6.1 模块3 内部结构与子模块协作关系

模块3（异常行为检测引擎）实现于 anomaly_engine.py 主文件中，检测器拆分为独立插件文件
（detectors/ 目录下 10 个插件）。逻辑上可分为两个子模块：基线学习器和一组插件化检测器。
两者通过 "基线建模 → 实时对比 → 偏差判定 → 后处理增强" 的闭环协作完成未知威胁发现。

【数据流全景】

与模块2并行运行——模块3同样通过 MessageBus 订阅模块1发布的 "traffic_record" 事件。
每条 TrafficRecord 经过白名单过滤后，进入双轨处理：
  ● 轨道一（基线更新）：提取统计特征，更新对应 IP 主机的行为基线模型
  ● 轨道二（异常判定）：将当前统计特征与基线对比，超出阈值则生成 anomaly_alert

模块3与模块2的关键区别在于：模块2依赖已知签名（找"是什么"攻击），
模块3依赖行为基线（找"是否异常"），两者互补覆盖已知 + 未知威胁。

【子模块1: 基线学习器】

该子模块负责为每台被监控的 IP 主机建立正常行为基线。基线是一系列统计指标的均值/方差，
描述了该主机在"正常"状态下的网络行为模式。

核心逻辑流程如下：

  (1) 基线初始化阶段（前60秒）
      ● 引擎启动后，默认进入基线学习期（60秒冷却期）
      ● 在此期间，所有流量仅用于累积统计数据，不产生任何告警
      ● 学习期结束后，基线模型建立完成，进入检测阶段

  (2) 主机追踪（_host_tracker 字典）
      ● 键：IP 地址（如 "192.168.1.100"）
      ● 值：HostProfile 对象，包含：
          - conn_timestamps：连接时间戳列表（滑动窗口维护）
          - unique_peers：对端 IP 集合（去重）
          - syn_count / total_count：SYN 包比例统计
          - bytes_sent / bytes_received：流量累计
          - unique_ports：访问的目标端口集合
          - last_seen：最近活跃时间

  (3) 基线指标计算
      学习期结束后，从 HostProfile 计算以下基线值并冻结：

      ● 连接速率基线
          - conn_rate_mean：每分钟平均连接数
          - conn_rate_std：连接数标准差
          - 计算方式：时间窗口内 (连接数 / 窗口时长) × 60

      ● 通信对端数基线
          - peer_count_mean：平均同时通信的对端 IP 数
          - 计算方式：时间窗口去重后的 unique_peers 大小

      ● SYN 比例基线
          - syn_ratio_mean：正常 SYN 包 / 总包比例
          - 用于检测 SYN Flood 攻击

      ● 端口访问基线
          - unique_ports_baseline：正常访问的端口种类数
          - common_ports：出现频率最高的端口列表

      ● 流量基线
          - bw_avg：平均带宽（字节/秒）

  (4) 基线持续更新
      ● 基线不是一成不变的——检测阶段仍以指数滑动平均（EWMA）方式缓慢更新
      ● 更新系数 alpha = 0.05，确保基线能适应网络环境的自然变化
      ● 被判定为异常的流量不计入基线更新（避免污染基线）

【子模块2: 异常检测器】

该子模块负责将每条新流量记录的实时统计与已建立的基线对比，判定是否存在异常行为。

核心检测器共有**十大类**（含 3 个后处理/增强型检测器）：

  (1) 端口扫描检测器（PortScanDetector，优先级 10）
      检测原理 … [不变]

  …（中间省略 SYN Flood(15)、带宽异常(18)、暴力破解(20)、异常外联(25)、横向扩散(30)、行为突变(40)，共 7 个核心检测器）

  (8) 异常时段检测器（TimeProfileDetector，优先级 35）★v2.0 新增
      检测原理：
      ● 维护每主机 24 小时 EMA 基线直方图
      ● 凌晨 0-6 点连接数 > 历史均值 + 2σ → 后处理阶段提升告警置信度（最多 +0.25）
      ● 告警描述追加 [夜间异常时段] 标记

  (9) 置信度评分器（ConfidenceScorer，优先级 85）★v2.0 新增
      检测原理：
      ● 后处理阶段运行，不产生新告警
      ● 为每条告警按攻击类型动态计算 0.0-1.0 置信度评分
      ● 7 种攻击类型的评分规则详见设计文档 9.3 节

  (10) 自学习误报抑制器（SuppressionLearner，优先级 88）★v2.0 新增
      检测原理：
      ● 订阅 GUI 告警状态变更：用户标记"误报"→ 提取 (src_ip, attack_type, dst_port) 签名
      ● 签名持久化到 data/suppression_rules.json，跨会话保留
      ● 同模式 ≥2 次标记 → 后处理阶段匹配该签名的告警 confidence -0.35

  (11) 攻击链构建器
      检测原理：
      ● 攻击链模型（Cyber Kill Chain，七阶段）：
          侦察(Recon) → 武器化(Weaponize) → 投递(Deliver) → 利用(Exploit)
          → 安装(Install) → C2(Command) → 目标达成(Action)

      构建逻辑：
      ● 每次新告警产生时，检查是否与已有攻击链的主机关联
      ● 同一 src_ip 或 dst_ip → 追加到该攻击链的对应阶段
      ● 同一主机 + 间隔 < 5 分钟 → 合并到同一攻击链
      ● 攻击链覆盖 >= 3 个阶段 → 标记为"严重"，推送到 GUI

【两个子模块的协作方式】

  anomaly_engine.process_traffic(record)
      │
      ├── 白名单检查 → 命中则 return []
      │
      ├── 更新基线学习器：
      │       ├── 更新 HostProfile（conn_timestamps, peers, syn_count, bytes）
      │       └── 若在学习期 → 仅累积，return []
      │       └── 若在检测期 → 以 EWMA 缓慢更新基线
      │
      ├── ★ 遍历所有检测器插件（for detector in self._detectors）：
      │       ├── PortScanDetector.process(record, now)    → List[Alert]
      │       ├── TimeProfileDetector.process(record, now)  → 更新小时基线
      │       ├── SynFloodDetector.process(record, now)     → List[Alert]
      │       ├── BandwidthAnomalyDetector.process(record, now) → List[Alert]
      │       ├── BruteForceDetector.process(record, now)   → List[Alert]
      │       ├── AbnormalOutboundDetector.process(record, now) → List[Alert]
      │       ├── LateralMovementDetector.process(record, now) → List[Alert]
      │       ├── BehavioralDeviationDetector.process(record, now) → List[Alert]
      │       └── 收集所有告警到 alerts 列表
      │
      ├── ★ 后处理阶段（for detector in self._detectors）：
      │       ├── ConfidenceScorer.post_process_batch(alerts)    → 动态计算 confidence
      │       ├── SuppressionLearner.post_process_batch(alerts)  → 匹配抑制规则降权
      │       └── TimeProfileDetector.post_process_batch(alerts) → 夜间增强
      │
      ├── 噪音过滤 → 短时间内相同类型去重
      │
      ├── 攻击链更新 → 将告警关联到攻击链
      │
      ├── 发布告警 → MessageBus.publish("anomaly_alert", alert)
      │
      └── 返回 alerts 列表
              │
              └── 所有告警同时推送到 MessageBus（供模块4）

关键设计：
  ● 十大检测器独立并行——每个检测器维护自己的滑动窗口状态，
    互不干扰，一条 record 可能同时触发多个检测器
  ● 后处理阶段统一增强——置信度评分、误报抑制、时段增强在告警生成后统一处理
  ● 插件化架构——新增检测器只需创建文件 + 注册，无需改动引擎核心代码
  ● 学习期机制确保基线代表"正常"，避免冷启动误报
  ● 双阈值配置调和"演示效果"与"生产可用性"的矛盾


================================================================================
  第七章  系统功能模块4 — 图形用户界面 (GUI)
================================================================================

7.1 模块4 内部结构与子模块协作关系

模块4（GUI 界面）实现于 main_window.py 单一文件中。虽然代码集中，
逻辑上可分为四个子模块：告警展示器、流量展示器、统计面板、交互控制。
四个子模块共享一个 MainWindow 类实例，通过 tkinter 控件直接交互，
并通过 MessageBus 与引擎层解耦通信。

【数据流全景】

模块4 是系统的"终端"——它同时订阅 MessageBus 的两个告警事件
（signature_alert 和 anomaly_alert）和一个流量事件（traffic_record），
将所有检测结果统一汇聚到 GUI 界面上展示。

模块4 与检测引擎的关系是纯粹的消息消费者——它不知道也不关心
告警来自模块2还是模块3，只关注呈现给用户的最终结果。
这种松耦合设计使引擎层的修改不影响界面层（反之亦然）。

【子模块1: 告警展示器】

该子模块负责告警列表的实时展示、筛选、详情查看和状态管理。

核心组件：

  (1) 告警表格（_tree: ttk.Treeview）
      ● 9 列：ID、时间、攻击类型、等级、来源、目标、描述、状态、备注
      ● 按严重度着色（通过 tkinter Treeview tags）：
          - 严重(5)：红色 (#f5222d)
          - 高危(4)：橙色 (#fa8c16)
          - 中危(3)：黄色 (#faad14)
          - 低危(2)：绿色 (#52c41a)
          - 信息(1)：蓝色 (#1890ff)
      ● 新数据追加到表格底部，超出上限（默认 1000 条）自动删除最旧记录

  (2) 告警筛选栏
      ● 三维组合筛选：IP 地址 + 攻击类型 + 端口
      ● IP 和端口输入框带下拉建议（自动从历史数据收集可选项）
      ● 点击"筛选"按钮 → 隐藏不匹配的行（而非删除）
      ● 点击"清除"按钮 → 恢复显示全部

  (3) 告警详情弹窗（双击触发）
      ● 双击表格行 → 弹出详情窗口，显示告警的全部字段：
          alert_id、时间戳、攻击类型/名称、严重度/置信度
          来源 IP:端口、目标 IP:端口
          匹配特征、载荷片段（文本框，可选中复制）
          处理建议
      ● 底部按钮：下载原始流量包（.bin/.json）

  (4) 告警处理状态（右键菜单）
      ● 右键任意告警行 → 弹出菜单：
          - 修改状态 → 待处理 / 已确认 / 误报 / 已忽略
          - 编辑备注 → 弹出文本框自由输入
      ● 状态和备注在内存的 _alert_states 字典中维护
      ● 重置时清空

  (5) 滚动行为控制
      ● 表格默认不自动滚动（新数据追加到底部，不改变视图位置）
      ● 仅当用户滚动条处于最底部时（yview[1] >= 0.99），
        新数据到达后自动 see(最后一行)，实现"跟随最新"
      ● 用户向上滚动查看历史数据时，视图不跳

【子模块2: 流量展示器】

该子模块负责正常流量的实时展示（与告警列表对称设计）。

核心组件：

  (1) 流量表格（_traffic_tree: ttk.Treeview）
      ● 7 列：#（序号）、协议、来源 IP:端口、目标 IP:端口、载荷摘要、流量大小
      ● 按协议着色：
          - TCP：蓝色
          - UDP：绿色
          - HTTP：橙色
          - DNS：紫色
          - 其他：默认
      ● 同样支持追加到底部 + 上限淘汰

  (2) 流量筛选栏
      ● 三维组合筛选：IP + 协议 + 端口
      ● 与告警筛选栏共享 _known_ips / _known_ports 数据集
      ● 筛选逻辑与告警栏相同（隐藏不匹配行）

  (3) 流量详情弹窗 + 下载
      ● 双击表格行 → 弹窗显示完整字段
      ● 有原始载荷 → "下载"按钮导出为 .bin 二进制文件
      ● 无载荷（仅元数据）→ 导出为 .json 结构化数据
      ● 弹出系统"另存为"对话框

【子模块3: 统计面板】

该子模块位于 GUI 右侧，实时展示检测系统的运行统计。

核心组件：

  (1) 统计指标
      ● 告警总数（大字体显示）
      ● 攻击类型分布（按数量排序的条形图区域）
      ● 严重度分布（1-5 级各数量）
      ● 攻击链数量
      ● 运行时间（启动后计时器）

  (2) 更新机制
      ● 每个 Alert 到达时触发 _update_stats()
      ● 攻击类型和严重度使用字典计数器维护
      ● 运行时间使用 root.after(1000, self._tick) 每秒更新

【子模块4: 交互控制器】

该子模块负责顶层菜单栏、工具栏按钮、状态栏和系统控制。

核心组件：

  (1) 菜单栏
      ● 文件菜单：导出告警 CSV、导出告警 TXT、导出报告 HTML、退出
      ● 配置菜单：检测阈值设置、白名单管理
      ● 帮助菜单：使用帮助（5标签页）、关于

  (2) 工具栏
      ● ▶ 开始检测 → 启动模块1抓包 + 模块2/3检测 + 模块4订阅
      ● ■ 停止检测 → 停止抓包，各引擎保持状态
      ● ↺ 重置统计 → 清空所有数据 + 通知各引擎重置

  (3) 状态栏
      ● 左侧：状态文本（就绪 / 运行中 / 错误提示）
      ● 右侧：运行时间（HH:MM:SS 格式）

  (4) 白名单管理弹窗
      ● IP 列表显示（Listbox + 滚动条）
      ● 手动输入添加
      ● 快速添加下拉框（从已出现的 IP 中选择）
      ● 删除选中
      ● 修改直接操作 config.WHITELIST_IPS 共享 list，引擎即时生效

  (5) 检测阈值配置弹窗（ConfigDialog）
      ● 以表格展示所有可调参数和当前值
      ● 双击单元格编辑数值
      ● 保存后发布 config_change 事件通知各模块

  (6) HTML 报告导出
      ● 收集当前告警数据 + 流量统计
      ● 生成完整 HTML 文件（内嵌 CSS，无外部依赖）
      ● 内容包括：概览卡片、检测结论、攻击类型/严重度分布条形图、
        告警处理状态、流量协议分布、告警明细表

  (7) 使用帮助弹窗
      ● 5 标签页 Notebook：快速开始、界面指南、筛选与标记、白名单与配置、导出报告
      ● 纯文本格式，无需外部文件

【四个子模块的协作方式】

四个子模块均在 MainWindow.__init__() 中初始化，通过 self 共享状态。

  (1) 引擎 → GUI 的数据流入：
      MessageBus 消息到达
          │
          ├── "traffic_record"  → self._on_traffic(record)
          │       ├── _insert_traffic_row()        [子模块2 展示]
          │       └── _update_known_ips_ports()    [子模块1/2 筛选建议]
          │
          ├── "signature_alert" → self._on_alert(alert)
          │       ├── _insert_alert_row()          [子模块1 展示]
          │       ├── _update_stats()              [子模块3 统计]
          │       └── _update_known_ips_ports()    [筛选建议]
          │
          └── "anomaly_alert"  → self._on_alert(alert)
             （同上，两类告警统一处理）

  (2) GUI → 引擎的控制流出：
      用户操作
          │
          ├── 点击"▶ 开始检测" → main.py 中 connect() 启停各模块
          ├── 点击"■ 停止检测" → main.py 中 disconnect() 停模块
          ├── 点击"↺ 重置"     → 清空GUI + MessageBus.publish("config_change")
          └── 白名单/阈值修改  → 操作 config 对象（共享引用，引擎即时生效）

  (3) 线程安全：
      ● MessageBus 回调在后台线程触发 → 所有 GUI 操作通过 root.after(0, ...)
        投递到 tkinter 主线程执行
      ● 避免 tkinter 多线程渲染导致的花屏/崩溃
